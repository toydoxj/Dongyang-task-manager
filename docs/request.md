## 날인요청 시나리오.

- 승인 단계 : 일반직원 -> 팀장 -> 관리자 승인 체계
- 팀장, 관리자도 날인요청이 가능함
- 파일저장 : Works Drive 프로젝트 폴더에 0.검토자료\YYYYMMDD\ 하위폴더에 저장함. (S3는 안씀) : 반드시 파일을 저장할 필요는 없음
- 검토구분 : 구조계산서, 구조안전확인서, 구조검토서, 구조도면, 보고서, 기타
- 날인요청은 반드시 프로젝트 상세 페이지에서만 가능함 (날인요청 버튼은 "날인취소" 버튼으로 변경, 버튼 우측에 현재 진행 단계를 표시함)
- 진행 단계 : 1차검토 중, 2차검토 중, 반려, 승인

### 구조계산서
- 입력 내용 : 실제출처(프로젝트 정보에 기입된 발주처와 다른경우에만 작성), 구조안전확인서 포함 (체크), Revision, 용도(예 : 허가용, 실시설계, 착공용 등), 비고
- 제목 : {code}_구조계산서_rev{N}_{용도}

### 구조안전확인서
- 입력 내용 : 실제출처(프로젝트 정보에 기입된 발주처와 다른경우에만 작성), 용도(예 : 허가용, 실시설계, 착공용 등), 비고
- 제목 : {code}_구조안전확인서_{용도}

### 구조검토서
- 입력 내용 : 실제출처(프로젝트 정보에 기입된 발주처와 다른경우에만 작성), 내용요약, 비고
- 제목 : {code}_{문서번호}_구조검토서
- 문서번호 : YY-의견-{NNN} (예 : 26-의견-056) , NNN은 당해 연도의 마지막 번호 다음 번호로 함.

### 구조도면
- 입력 내용 : 실제출처(프로젝트 정보에 기입된 발주처와 다른경우에만 작성), 용도(예 : 허가용, 실시설계, 착공용 등), 비고
- 제목 : {code}_구조도면_{용도}

### 보고서
- 입력 내용 : 실제출처(프로젝트 정보에 기입된 발주처와 다른경우에만 작성), 비고
- 제목 : {code}_보고서

### 기타 
- 입력 내용 : 실제출처(프로젝트 정보에 기입된 발주처와 다른경우에만 작성), 문서종류, 비고
- 제목 : {code}_{문서종류}

### Works의 Bot API 활용
- 날인요청 제출 시 팀장에게 알림
- 팀장이 반려를 하면 반려사유와 함께 일반직원에게 알림
- 팀장이 승인을 하면 관리자에게 알림
- 관리자가 반려를 하면 반려사유와 함께 일반직원에게 알림
- 팀장이 날인요청을 하는 경우 관리자에게 알림

### 팀장과 관리자는 날인요청 페이지에서 내용 확인이 가능함 (일반직원은 접근 불가)

### 날인요청 페이지 권한 / 정렬 (2026-05-01)

- **팀장**: 자기 팀 진행 현황만 표시. 본인 이름의 `Employee.team`을 기준으로
  같은 팀 active 직원이 요청자인 건만 노출.
- **관리자**: 전체 현황을 표시.
- **상태 정렬 우선순위 (1차 정렬)**
  - 팀장: `1차검토 중` → `2차검토 중` → `반려` → `승인`
  - 관리자: `2차검토 중` → `1차검토 중` → `반려` → `승인`
- **상태 안 정렬 (2차 정렬)**
  - 검토중·반려: 제출예정일 가까운 것이 먼저 (asc, 빈 값은 마지막)
  - 승인: 제출예정일 최신 순 (desc — 최근 승인이 위)

### 자동 TASK 생성 / 동기화 (2026-05-01)

날인요청 등록 시 노션 TASK DB에 자동으로 row를 생성해 본인/검토자
일정에 반영한다. 생성된 TASK page_id는 SealRequest의 별도 컬럼에 저장:

| 컬럼 | 용도 | 담당자 |
|---|---|---|
| `연결TASK` | 요청자 본인용 | 요청자 |
| `1차검토TASK` | 1차 검토자(팀장 또는 admin)용 | 등록 시점에 결정 |
| `2차검토TASK` | 2차 검토자(admin)용 | 1차 승인 시점에 생성 |

**시작/완료일 정책**
- TASK 시작일 = 해당 단계가 시작된 날
  - 요청자/1차검토 TASK: 등록일
  - 2차검토 TASK: 1차 승인일
- TASK 완료일 = 해당 단계가 끝난 날 (별도 컬럼 X — `기간.end` 채움)
  - 1차검토 TASK: 1차 승인일 또는 1차 반려일
  - 2차검토 TASK: 최종 승인일 또는 2차 반려일
  - 요청자 TASK: 최종 승인일 (반려 시엔 그대로 진행 — 재요청 가능하므로)

**라이프사이클**
1. 등록 → 요청자 TASK + 1차검토 TASK (둘 다 시작일=등록일, 진행 중)
2. 1차 승인 → 1차검토 TASK 완료(end=1차승인일) + 2차검토 TASK 생성(start=1차승인일)
3. 2차 승인 → 2차검토 TASK 완료(end=승인일) + 요청자 TASK 완료(end=승인일)
4. 반려 → 현재 단계 검토자 TASK 완료(end=반려일). 요청자 TASK는 진행 중 유지.
5. 재요청(반려→1차검토 중) → 1차검토 TASK 새로 생성(이전은 완료된 상태)
6. 날인취소 → 모든 연결 TASK를 `완료`로 자동 마감(end=취소일). archive 아님 — 흔적 보존.

### 재요청
- 반려가 되면 프로젝트내 날인요청 버튼은 날인재요청으로 변경됨 (날인취소 버튼도 생성되어 취소도 가능함)
- 날인재요청시 이전에 기입된 내용이 그대로 나타남(수정가능)
- 이후 절차는 동일함

### 날인취소 시
- 날인취소 버튼을 누르면 알림을 띄워줘야함.
- 날인취소가 되면 노션의 내용은 삭제됨
- 구조검토서의 경우 생성된 {문서번호} 뒤에 번호가 생성되어 있으면, 노션에서 "날인취소" 표기를 함. (중간에 문서번호 누락 방지, 더 좋은 아이디어가 있으면 제안 가능)
