# 진행률 평가 방식 — 분석 및 제안

> 양식 개선 전에 해결해야 할 **가장 근본적인 질문**:
> "240708 오뚜기 안양공장 80%"의 80%는 도대체 무엇을 의미하는가?

---

## 1. 현행 진행률의 문제 진단

### 1.1 데이터 수집 방식 — 사람이 손으로 적는 숫자

업무일지의 진행률 컬럼은 **PM이 주관적으로 추정한 숫자**다. task-manager 코드 확인 결과:

| 위치 | 진행률 산출 방식 |
|---|---|
| `mirror_projects.stage` | 사람이 select (진행중/대기/보류/완료/타절/종결/이관) |
| `mirror_projects` 자체 | **별도 진행률 컬럼 없음** — 업무일지의 % 숫자는 노션의 어디서 오는가? |
| `mirror_tasks.progress` | 사람이 0~1 사이 숫자 입력 (자동 산출 아님) |
| `task_auto_progress.py` | 시작일 도래 시 status를 "시작 전 → 진행 중"으로만 바꿈 (진행률 미변경) |

**결론**: 진행률 80%는 **PM의 직관적 추정치**임. 객관적 산출 근거 없음.

### 1.2 현행 데이터의 모순들 (업무일지에서 직접 추출)

| CODE | 단계 | 진행률 | 모순 |
|---|---|---|---|
| 220104 | 보류 | 80% | 보류인데 80%? 진행 중단인지 거의 끝난 건지 불명 |
| 230502 | 실시설계 | **100%** | 100%인데 왜 보고가 계속 됨? |
| 220302 | 보류 | 0% | 시작도 안 했는데 보류? = 발주처 회신 대기? |
| 260108 | 기본설계 | **100%** | 기본설계 100%면 실시설계로 넘어가야 함 |
| 250310 | 실시설계 | **100%** | 동일 |
| 200504 | 기본 및 실시설계 | 99% | 99%가 6개월째? |

**진단**: 진행률이 **단계 변경의 트리거가 되지 못함**. 100%여도 단계가 안 넘어가고, 보류여도 진행률이 그대로 유지됨.

### 1.3 진행률이 의미를 잃는 3가지 원인

| 원인 | 설명 | 예시 |
|---|---|---|
| **기준의 부재** | "80%"가 무엇 대비 80%인지 정의 안 됨 — 시간? 도면 매수? 작업량? 비용? | PM마다 다른 의미로 사용 |
| **이산값 사용** | 10% / 50% / 80% / 90% / 95% / 100% 같은 **5~10단계 자릿수만 사용** — 연속 측정이 아니라 사실상 categorical | "10% 단계"는 "착수했다", "90% 단계"는 "거의 끝났다" 의미 |
| **갱신 누락** | 한번 80% 적으면 한참 80%에 머물러 있다가 마감 직전 95% → 100%로 점프 | 위 230502, 260108 사례 |

## 2. 진행률을 정의하는 4가지 방식

업계 일반론으로 진행률 산출에는 4가지 방법이 있다.

| 방식 | 정의 | 객관성 | 동양구조 적용 가능성 |
|---|---|---|---|
| **A. 시간기반** | 경과일 ÷ 총 계약기간 | 매우 높음 | 단순하나 작업량 반영 못함 |
| **B. 마일스톤** | 단계별 가중치 합 (계획설계 20%/기본 40%/실시 80%/완료 100%) | 높음 | **권장 — 구조설계 표준 단계와 정합** |
| **C. 산출물(EV)** | 계약 산출물 중 완성·승인된 비율 | 매우 높음 | 도면 매수·문서 수 정의 필요 |
| **D. 비용기반(Earned Value)** | 투입원가 ÷ 예상총원가 | 높음 | 외주비·인건비 추적 필요 (현재 미구현) |

### 2.1 권장: B (마일스톤) + C (산출물) 하이브리드

구조설계의 표준 단계를 기반으로 **사전 정의된 가중치**를 두되, 산출물 체크가 들어가면 보정하는 방식.

```
계획설계   →  기본설계   →  실시설계   →  납품   →  현장지원   →  종결
 (0~20%)     (20~40%)    (40~80%)    (80~95%)   (95~99%)   (100%)
```

각 단계 진입 시 **진행률 하한**이 자동 설정되고, 단계 내에서는 산출물 진척도로 세분 — PM 입력 부담 최소화.

## 3. 동양구조 표준 단계 모델 제안

### 3.1 단계별 가중치 정의

업무일지에서 추출한 실제 단계 명칭을 정리하면:

| 노션 stage | 권장 가중치 (시작 → 완료) | 비고 |
|---|---|---|
| 견적/입찰 | 0% (영업 단계) | 별도 sales DB |
| 수주확정 | 5% | 계약체결 시 |
| 계획설계 | 5 → 20% | 평면·구조형식 결정 |
| 기본설계 | 20 → 40% | 부재 단면, 해석모델 1차 |
| 중간설계 | 40 → 60% | (필요 시) — 일부 프로젝트만 |
| 실시설계 | 40 → 85% | 도면·계산서·내역서 |
| 납품/심의 | 85 → 95% | 도서 제출 + 심의 대응 |
| 현장지원 | 95 → 99% | 시공 중 RFI 대응 |
| 종결 | 100% | 완료보고서 |
| 보류 | 멈춤 (값 유지) | 이전 단계 진행률 그대로 |
| 타절 | -1 (중단) | 별도 표시 |

> 이 표가 **회사 표준**이 되어야 한다. 한 번 합의되면 모든 PM이 동일 의미로 사용.

### 3.2 단계 내 세분화 (산출물 체크리스트)

각 단계에서 **3~5개의 체크 항목**으로 진행률을 세분.

**예시: 실시설계 (40 → 85%, 폭 45%)**

| 체크항목 | 가중치 (단계 내) | 절대 진행률 |
|---|---|---|
| ☐ 구조해석 모델 완성 | 20% | 40 → 49% |
| ☐ 부재 검토 (보·기둥·슬래브) | 25% | 49 → 60% |
| ☐ 도면 작성 (구조평면·일반·상세) | 30% | 60 → 73% |
| ☐ 계산서 작성 | 15% | 73 → 80% |
| ☐ 도면 검토(QA) | 10% | 80 → 85% |

**산출 공식**:
```
프로젝트 진행률 = stage_start + (stage_end - stage_start) × (체크된 가중치 합 / 100)
```

체크 시점에 자동 갱신 → **PM 추정 의존도 제거**.

## 4. 데이터 모델 변경안

### 4.1 신규 테이블: `project_milestones`

```python
# backend/app/models/milestone.py
class ProjectMilestone(Base):
    __tablename__ = "project_milestones"
    
    id: Mapped[int] = mapped_column(primary_key=True)
    project_id: Mapped[str] = mapped_column(ForeignKey("mirror_projects.page_id"))
    stage: Mapped[str]              # 계획설계/기본설계/실시설계/납품/현장지원
    item_name: Mapped[str]          # "구조해석 모델 완성"
    weight: Mapped[float]           # 단계 내 가중치 (합=100)
    is_completed: Mapped[bool] = mapped_column(default=False)
    completed_at: Mapped[datetime | None]
    completed_by: Mapped[int | None] = mapped_column(ForeignKey("users.id"))
    sort_order: Mapped[int]
    
    # 단계 정의는 별도 테이블 또는 enum
```

### 4.2 단계 정의 — 코드 vs DB

**선택지**:

| 방식 | 장점 | 단점 |
|---|---|---|
| 코드 상수 (enum) | 변경 시 마이그레이션 불필요, 일관성 | 표준 변경 시 배포 필요 |
| DB 테이블 (`stage_templates`) | admin이 UI에서 수정 가능 | 복잡도 증가 |

**권장**: 초기 **코드 상수**로 구현. 1년 운영 후 표준이 안정되면 DB로 이전.

```python
# backend/app/services/progress_calculator.py
STAGE_WEIGHTS = {
    "수주확정": (0, 5),
    "계획설계": (5, 20),
    "기본설계": (20, 40),
    "실시설계": (40, 85),
    "납품/심의": (85, 95),
    "현장지원": (95, 99),
    "종결": (100, 100),
}

DEFAULT_MILESTONES = {
    "실시설계": [
        ("구조해석 모델 완성", 20),
        ("부재 검토 (보·기둥·슬래브)", 25),
        ("도면 작성 (평면·일반·상세)", 30),
        ("계산서 작성", 15),
        ("도면 검토(QA)", 10),
    ],
    # ... 다른 단계
}

def calc_progress(project_stage: str, milestones: list[ProjectMilestone]) -> float:
    """단계별 가중치 + 마일스톤 체크 → 절대 진행률 산출."""
    if project_stage not in STAGE_WEIGHTS:
        return 0.0
    start, end = STAGE_WEIGHTS[project_stage]
    if not milestones:
        return start  # 단계 입장만, 마일스톤 없음
    completed_weight = sum(m.weight for m in milestones if m.is_completed)
    total_weight = sum(m.weight for m in milestones)
    if total_weight == 0:
        return start
    return start + (end - start) * (completed_weight / total_weight)
```

## 6. 마일스톤 데이터 채우기 전략

### 6.1 신규 프로젝트 — 자동 생성

`POST /api/projects` 시 단계별 `DEFAULT_MILESTONES` 자동 복제:

```python
async def create_project(req: ProjectCreateRequest):
    project = await notion.create_project(req)
    # 모든 단계의 디폴트 마일스톤 일괄 생성 (각 단계 5개씩 × 5단계 = 25개)
    for stage, items in DEFAULT_MILESTONES.items():
        for name, weight in items:
            db.add(ProjectMilestone(
                project_id=project.id,
                stage=stage,
                item_name=name,
                weight=weight,
                is_completed=False,
            ))
    db.commit()
```

## 7. 진행률의 다른 차원 — "건강도(Health)" 추가

진행률 한 숫자만으로는 부족. **3가지 신호**를 함께 표시:

| 신호 | 정의 | 색상 |
|---|---|---|
| **진행률** (Progress) | 위에서 정의한 마일스톤 기반 % | 회색 막대 |
| **속도** (Velocity) | Δ주간 진행률 — 정체 여부 | +값 녹색 / 0 회색 / -값 빨강 |
| **여유도** (Slack) | (남은 진행률) ÷ (남은 기간) — 마감 대비 충분한가 | 초록(여유)/노랑(빠듯)/빨강(부족) |

**예시**:
```
240708 │ 오뚜기 안양공장 │ 실시설계 │ 64.5% │ +5%/주 │ ⬜OK     D-21
241105 │ 영등포농협      │ 계획설계 │ 10.0% │ +0%/주 │ 🟥부족   D-7
```

> 같은 진행률 10%여도 **여유도가 다르면 신호가 달라짐**. PM의 우선순위 판단 자동화.

## 8. 안전장치

### 8.1 진행률 후퇴 방지

마일스톤 체크 해제는 admin만 가능 (PM은 한번 체크하면 되돌릴 수 없음). 사후 변경은 별도 로그로 추적.

### 8.2 신뢰도 다운 방지

기존 PM들이 자기 직관과 다른 산출 결과를 받으면 시스템 신뢰가 떨어짐. **운영 첫 3개월은 두 진행률을 병기**:

```
240708 │ 오뚜기 안양공장 │ 실시설계 │ 64.5% (PM: 80%)
```

차이가 큰 프로젝트는 PM과 검토 → **마일스톤 정의 자체를 보정**(원인이 정의에 있는 경우).

### 8.3 프로젝트별 커스텀 마일스톤

표준 마일스톤이 안 맞는 경우(예: 안전진단, 구조검토 단건):

- 단계 정의는 표준 유지
- 마일스톤 항목만 추가/제거 허용
- 가중치는 합 100% 강제

## 10. 결정 필요 사항

가장 중요한 것은 **단계 가중치 합의**. 다음을 확정해야 다음 단계 진행 가능:

1. **회사 표준 단계가 7개(`수주확정/계획/기본/실시/납품/현장/종결`)로 충분한가?** 추가 단계 필요 여부?
2. **각 단계별 가중치 비율** (위 §3.1 표는 안일 뿐) — 실제 내부 노력 분포와 맞는가?
3. **마일스톤 정의** — 단계별 5개 디폴트 항목을 누가 정할지 (구조1팀장? 임원회의?)
4. **기존 프로젝트 처리** — 일괄 마이그레이션 vs 신규 프로젝트만 적용?
5. **이중 표시 기간** — PM 추정값과 산출값 병기할 것인가, 처음부터 산출값만?

## 11. 핵심 메시지

> 양식 개선의 모든 시각화(진행률 바, Δ 표시, 위험 페이지)는 **진행률 숫자가 신뢰할 수 있을 때만 의미가 있다**. 

현재처럼 PM 직관에 의존한 80%·90%·95%·100% 점프 패턴이 유지되면, **아무리 양식을 개선해도 결국 추정 숫자를 더 보기 좋게 표시할 뿐**임.

따라서 작업 순서는:

```
[1] 진행률 산출 표준 합의 (조직적 결정)
        ↓
[2] 마일스톤 시스템 구축 (개발)
        ↓
[3] 데이터 마이그레이션 + PM 검증 (운영)
        ↓
[4] 양식 개선 적용 (개발 + 검증)
```

진행률 산출 표준 합의([1])가 안 되면 [4]는 의미가 없음. **양식 개선보다 먼저 회사 차원의 단계 정의 합의가 필요**함.
