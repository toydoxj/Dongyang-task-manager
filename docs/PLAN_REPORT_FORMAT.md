## 1. 현행 양식의 한계 진단

현행 업무일지는 **Excel 셀 병합 기반의 정적 표**로, 정보 표현에 다음 한계가 있다:

| 한계 | 구체적 문제 |
|---|---|
| **■ 마크의 모호성** | 담당자별 ■ 표시는 "참여 여부"만 보여주고 **참여 강도(주역/보조), 활동 유형(외근/사무실), 시간 비중**을 구분 못함 |
| **금주예정사항 자유텍스트** | 텍스트만 있어 **검색·집계·알림 트리거 불가**. "도면 작성"이 5명에게 흩어져 있어도 자동 집계 안 됨 |
| **진행률 단일 숫자** | "10%"만 표시 — **지난 주 대비 변화량(Δ)이 안 보임**. 정체된 프로젝트 식별 어려움 |
| **단계 구분 부재** | "보류 100%", "보류 0%" 같은 모순적 표시 다수. **단계와 진행률의 관계가 명확치 않음** |
| **개인 일정 매트릭스의 빈약함** | 외근/연차/파견만 표시되고 **무엇을 위한 외근인지, 어느 프로젝트 출장인지 연결 안 됨** |
| **위험 신호 부재** | 마감 임박/지연/장기 정체 프로젝트가 **시각적으로 구분 안 됨** — 32개 프로젝트가 모두 같은 회색 글자 |
| **영업과 진행 프로젝트 단절** | 영업 → 수주 전환 시 어떤 영업 건이 어떤 프로젝트가 되었는지 **추적 불가** |
| **PM 워크로드 비가시화** | PM별로 몇 개 프로젝트 담당하는지, 부하가 어느 정도인지 **표에서 안 보임** |
| **읽는 사람 관점 부재** | 모두에게 같은 4페이지 — **사장은 위험·수주 신호, PM은 자기 팀, 담당자는 자기 일이 필요한데 동일 양식** |

## 2. 개선 원칙

| 원칙 | 의미 |
|---|---|
| **시그널 우선** | 32개 프로젝트가 균질하게 보이지 않게 — 위험·수주·완료·정체를 시각적으로 구분 |
| **델타(Δ) 표시** | 진행률, 인원, 영업 모두 **지난 주 대비 변화**를 함께 표시 |
| **Task 메타데이터 활용** | 난이도/우선순위/활동을 ■ 마크 대체에 활용 |
| **계층적 디테일** | 1페이지 = Executive Summary, 2~4페이지 = 팀별 디테일 (현행은 모두 디테일) |
| **링크 가능성** | PDF/xlsx여도 노션·앱으로 점프할 수 있는 QR/URL 컬럼 |
| **읽는 사람별 변형** | admin/PM/member별로 다른 정렬·하이라이트 |

## 3. 페이지별 개선안

### 3.1 1페이지 — Executive Summary 강화

#### A) 인원현황 → "이번 주 인원 변동" 추가

**현행**:
```
■ 인원현황 - 총원 : 32인  ■ 직종별인원 - 구조설계 : 22인, 안전진단 : 6인, 관리및세무 : 3인
```

**개선안**:
```
■ 인원현황 (2026-04-27 기준)
  총원 32인  │  구조설계 22 │ 안전진단 6 │ 관리세무 3
  변동: 신규 0  │  퇴사 2 (이종심 이사·황호준 기사)  │  순증 -2

■ 출근율 (이번 주)  92%  (휴가/외근 제외 사무실 출근일 기준)
```

> task-manager의 `category` (휴가/외근/출장) + `Employee.resigned_at` 집계로 자동 계산 가능. **부장급 이상이 가장 먼저 보는 숫자가 인원 변동임**.

### 3.2 2~4페이지 — 팀별 진행 상황 강화

#### F) ■ 마크 → "역할 + 활동" 표기

**현행** (담당자 5명 모두 ■):
```
240708  │ 오뚜기 안양공장 │ 계룡건설 │ 서동균 │ 80% │       │       │  ■   │       │       │
```

**개선안** (역할/활동 구분):
```
240708  │ 오뚜기 안양공장 │ 계룡건설 │ 서동균 │ 80% +5% │  PM   │       │ 🏢현장 │       │       │
        │                 │          │        │         │       │       │       │       │       │

범례: PM = 책임자, 🏢현장 = 현장근무, 🖥️사무실 = 설계작업, 📞협의 = 발주처 협의 주
```

> task-manager의 `task.activity` (사무실/외근/출장)과 `task.assignees` 첫 번째 = PM 컨벤션 활용. **이상호 과장이 5개 프로젝트에 모두 ■인지, 그 중 PM은 1개 인지**가 보임.

#### G) 진행률 → 현재값 + 델타(Δ) + 마감 거리

**현행**:
```
240708  │ 오뚜기 안양공장 │ 실시설계 │ 80% │
```

**개선안**:
```
240708  │ 오뚜기 안양공장 │ 실시설계 │  80%  ━━━━━━━━━━━━━━━━░░░░  +5% / 주
                                              ─────────────────
                                              마감 D-14  ⚠ 정상속도 OK
```

> 한 줄 레이아웃이 어렵다면 별도 컬럼:

```
CODE     │ 프로젝트          │ 단계      │ 진행 │ Δ주간 │ D-day │ 상태신호
240708   │ 오뚜기 안양공장   │ 실시설계  │ 80%  │ +5%   │ -14   │ ⬜정상
240901   │ 24-D-00부대       │ 기본설계  │ 10%  │ +0%   │ -45   │ 🟨정체(3주)
241105   │ 영등포농협        │ 계획설계  │ 10%  │ +0%   │ -7    │ 🟥지연
```

> **'정체'는 3주 이상 진행률 변화 없을 때, '지연'은 마감일 가까운데 진행률 낮을 때 자동 분류**. 32개 프로젝트가 더이상 균질해 보이지 않게 됨.

#### H) 단계 일관성 — "보류 0%" 모순 정리

**현행 데이터 모순**:
```
220302 │ 동두천 생연동 가로주택정비사업 │ 보류 │ 0%
260108 │ 역삼동 814번지 건물 대수선     │ 기본설계 │ 100%   ← 100%인데 기본설계?
220105 │ 고양장항 A6BL                 │ 보류 │ 80%    ← 80%인데 보류?
```

**개선안**: 단계는 4분면으로 표시.

```
┌──────────────────────────────────────────────┐
│  [완료]   240708 오뚜기  80%  ━━━━━━━━━░░    │  ← 진행중·정상
│  [완료]   220105 고양장항 80% ━━━━━━━━░░ ⏸  │  ← 진행중·일시정지
│  [대기]   220302 동두천   0%  ░░░░░░░░░░    │  ← 미착수
│  [완료]   260108 역삼동  100% ━━━━━━━━━━ ✓  │  ← 완료
└──────────────────────────────────────────────┘
```

> task-manager의 `stage` (진행중/대기/보류/완료/타절/종결/이관)과 `progress`를 **2D 매트릭스**로 시각화. 보류 표시는 `stage`로, 진행 정도는 `progress`로 분리하여 모순 제거.

```
│ ⑧ 지연 프로젝트 (마감 D-day < 14, 진행률 부족)                   │
│ ⑨ 정체 프로젝트 (3주+ 진행률 무변동)                             │
│ ⑩ 마감 임박 (D-day < 14, 정상)                                  │
│ ⑪ PM별 워크로드 차트                                             │
└────────────────────────────────────────────────────────────────┘
```

### 3~5페이지 — 팀별 상세 (개선)
```
┌────────────────────────────────────────────────────────────────┐
│ 구조1팀                                                         │
│  팀 헤더: PM별 부하 / 평균 Δ진행률                              │
│  프로젝트 표: CODE/이름/단계/진행률(+Δ)/D-day/신호/PM/금주예정   │
│  정렬: 신호(지연→정체→정상) → 진행률 내림차순                   │
└────────────────────────────────────────────────────────────────┘
```

> **현행 4페이지 → 개선 5페이지** (위험 페이지 신규). 정보 밀도는 비슷하지만 **시그널 중심으로 재배치**.

## 7. 주간 스냅샷 — Δ 계산을 위한 필수 인프라

진행률 Δ 표시를 위해 **매주 일요일 23:59 자동 스냅샷**이 필수.

```python
# backend/app/services/weekly_snapshot.py
"""매주 일요일 23:59 KST: 모든 mirror_projects/tasks의 진행률·단계 스냅샷 저장."""

class ProjectSnapshot(Base):
    __tablename__ = "project_snapshots"
    project_id: Mapped[str]
    week_start: Mapped[date]  # 해당 주 월요일
    progress: Mapped[float]
    stage: Mapped[str]
    assignees: Mapped[list[str]]
    snapshot_at: Mapped[datetime]
```

> 이 인프라가 있으면 **모든 Δ 표시가 단순 SQL JOIN으로 해결**됨. 향후 4주 추세, 12주 추세 같은 확장도 동일 테이블로 가능.

## 8. 의사결정 요청

다음 항목은 진행 전 결정 필요:

1. **양식 형식** — 현행 양식 유지(개선만) vs 완전 재설계? (권장: **단계적 개선** — P0만 먼저 적용해서 임원 반응 확인)

3. **■ 마크 폐기** — 즉시 폐기 vs 병행 표시? (권장: **3개월 병행** 후 폐기)
4. **Δ 표시 기준** — 1주 전 vs 4주 평균? (권장: **1주 전** — 단순함)
5. **금주예정사항 sub-task 전환** — 즉시 vs 운영 안정화 후? (권장: **운영 안정화 후 P2** — 사용자 학습 시간 필요)
